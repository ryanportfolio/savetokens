If you only do three things

  1. Compact early, then cut over to a fresh session. Run /compact once the context window is about 40 percent full, well before the limit. If you find yourself compacting a second time, start a new session instead and hand the work over with a handoff-audit prompt so the fresh session can verify it independently.
  2. Run /clear between unrelated tasks. Anything left over gets reread on every turn that follows.
  3. Shrink what the agent reads and writes. Name file paths instead of pasting files, and turn on caveman mode for shorter replies.

Why long sessions get expensive

The model has no memory between turns. Every time you send a message, the whole conversation goes with it: your prompts, the agent's replies, and every line of tool output. A token written on turn one is read again on every later turn.

So the cost of a session compounds as it grows. Ten turns on a bloated transcript cost far more than ten on a fresh one, because each turn pays again for everything before it. Prompt caching can cut the billed price of the reread, but the tokens still fill the window and still dilute the model's attention.

The budget is finite too: roughly a million tokens of context on current Opus and Sonnet models, per the vendor spec above, and a bigger window also means a bigger bill for a full one. Push close to the limit and older context gets dropped, not summarized, and answer quality falls before you hit the wall.

The basics: clear, compact, or start fresh

Three moves control how much history you carry. They differ in how much they drop, and picking the right one is most of the skill.

/clear Between unrelated tasks

Wipes the conversation, keeps the session open. Cheapest move there is. If the next task shares nothing with the current one, nothing is lost.

/compact At a checkpoint

Replaces the transcript with a summary. Use it after a fix lands, before a new area: you keep the conclusions, not the word-for-word history. Avoid it mid-investigation, because summaries drop detail you still need.

new session When the task changes

The cleanest cut. Steering a long transcript into new work makes every future turn pay rent on everything already said. Start fresh instead.

Alongside those, two habits shrink what enters the context in the first place:

  • Never paste whole files. Name the path and let the agent read the ranges it needs. The agent reads once; a pasted file gets reread every turn.
  • Ask precisely. "Fix the expiry check in auth middleware" sends the agent straight to one file; "look at the auth system" buys a tour of all of them.

The five-minute cache window

Prompt caching is the reason the reread on every turn does not bill at full price. The provider keeps the already-processed conversation in a cache: the first turn writes it at about 1.25x the normal input price, and every later turn reads it back at roughly a tenth of the normal price. Those figures are Anthropic's published caching rates, not measured here.

The catch is timing. The cache lives for about five minutes, and the clock resets every time it is used. Reply within the window and each turn's reread stays at the cheap rate. Walk away longer than that and the cache expires: the next message rewrites the entire transcript at the premium rate, as if the whole conversation were being sent for the first time. For the same tokens, that cold rewrite costs roughly twelve times a cached reread, so on a long transcript one idle gap can out-bill several active turns.

  • Work in bursts. While a session is hot, keep replies coming inside the five-minute window. Every turn that lands in the window pays a tenth of the price for the history it carries.
  • A long break is a decision point. Stepping away from a big transcript for more than a few minutes? That is the natural moment to /compact or /clear. Coming back cold pays the full rewrite either way, so come back to a small transcript instead of a large one.
  • Short transcripts make cold starts cheap. Every habit on this page that keeps the conversation small also shrinks the penalty when the cache does expire. The two compound.

Agent tools set the cache markers themselves; there is nothing to configure. Timing and transcript size are the levers you actually hold. Some setups use a one-hour cache at a higher write price, and the same logic applies with a longer clock.

Caveman mode: shorter replies, same facts

Caveman mode is a standing instruction that makes the agent's replies drastically shorter without losing technical content. The agent drops articles, filler, pleasantries, and hedging, speaks in fragments, and uses arrows for causality. Code, commits, file contents, function names, and error strings are never touched.

The target is estimate~50% less reply prose, deliberately lowballed. The spent side is logged: caveman replies total 64.7 million tokens across 615 sessions. The plain-prose baseline is assumed at 2x, which puts the estimated saving at roughly 65 million tokens.

SettingWhat changesExample: why does my React component re-render
lite No filler or hedging. Full sentences kept. Professional but tight "Your component re-renders because you create a new object reference on each render. Wrap it in useMemo."
full Articles dropped, fragments allowed, short synonyms "New object ref each render. Inline object prop = new ref = re-render. Wrap in useMemo."
ultra Common words abbreviated, arrows for causality, one word when one word is enough. Code names never abbreviated "Inline obj prop → new ref → re-render. useMemo."

The rule that makes it safe: accuracy first, brevity second. A fact, caveat, or qualifier is never dropped to save tokens; only the wording compresses.

The agent also switches back to plain prose on its own for security warnings, irreversible-action confirmations, and any step where compressed wording could be misread.

Caveman mode 
## Response style: caveman (ultra)

Respond terse like smart caveman. All technical substance stays; only filler goes.
Accuracy first, brevity second. Never drop a fact, caveat, or qualifier to save
tokens: compress wording, not meaning. If terse risks a misread, spend the words.

Drop: articles (a/an/the), filler (just/really/basically/actually/simply),
pleasantries (sure/certainly/of course), hedging. Fragments OK. Short synonyms
(big not extensive, fix not "implement a solution for"). Technical terms exact.
Code blocks unchanged. Error messages quoted exact.

Ultra intensity: abbreviate prose words (DB/auth/config/req/res/fn/impl), use
arrows for causality (X -> Y), one word when one word is enough. Code symbols,
function names, API names, and error strings are never abbreviated.

Auto-clarity: switch to plain prose for security warnings, irreversible-action
confirmations, and multi-step sequences where fragment order could be misread.
Resume caveman after the clear part is done.

Boundaries: code, commits, PR text, and file contents are written normally.
Off switch: "stop caveman" / "normal mode".

Want it gentler? Delete the ultra paragraph for classic caveman, or keep full sentences for the lite setting shown in the table above.

Context discipline rules 
## Context discipline

- One task per session. Unrelated task -> /clear or a new session.
- Compact at natural checkpoints (after a fix lands, before a new area),
  never mid-investigation.
- Do not paste whole files into the prompt. Name the path and let the agent
  read targeted ranges.
- Keep this file a thin kernel. Topical detail lives in reference files the
  agent loads on demand.
- Subagents are not a discount. A subagent rereads context fresh and pays a
  summarize-back tax; dispatch only for large, genuinely independent scopes.
- Prefer filtered command output over raw dumps (diffs, logs, test runs).

The last rule is where output filtering plugs in; the measured numbers behind it are on the measurements page.

Handoff audit as a skill 
---
name: handoff-audit
description: >
  Draft a self-contained prompt for a fresh session to independently
  verify the current change. The prompt is the only deliverable.
---

When the user hands off work for verification, write ONE fenced markdown block
they can paste into a new session that has zero shared context. Do not run the
audit here or spawn a subagent: independence is the whole point.

Scope is the work on this branch. Capture the repo URL, the branch, and the
commit SHAs (git log --oneline main..HEAD, git diff --stat main..HEAD) verbatim.

Build the prompt from:
- Claims: what changed, what must still hold (the invariants), and why each
  risky call was safe, so the auditor can try to falsify each one.
- Fidelity reference: for a move or refactor, the pre-change git handle
  (git show <base-sha>:path) so they diff old against new instead of
  trusting a summary.
- Checks, each with an explicit pass/fail: no dangling consumers of moved or
  deleted symbols (grep -rw; warn about substring and pointer-comment false
  positives), deleted code was really dead, kept code is still wired, every
  reference site was updated, syntax balances.
- Constraints: what this environment cannot verify (installs, type-checks, a
  dev server) and the real signal that does (CI, deploy log, test suite).
- Output contract: PASS/FAIL per check with file:line evidence, re-verify each
  FAIL before reporting it, fix only real breaks, commit only touched files.

Open with an adversarial role ("skeptical auditor; assume the change is wrong
until your own grep and read prove otherwise") and a grounding line ("ground
every verdict in files you open yourself; do not trust this prompt's summary").
Wrap each part in its own tag: <audit_target>, <claims>,
<fidelity_reference>, <checks>, <constraints>,
<output_contract>.

Save it as .claude/skills/handoff-audit/SKILL.md (or under .agents/skills/ for Codex). Invoke it when you cut a task over to a fresh session: it writes the verification prompt, it does not run the audit itself.

Advanced: models, subagents, workflows

Once the basics are habit, the next savings come from routing work to the right place. These matter most on big projects and long working days.

Audit before you optimize

Look at where the tokens actually go before changing habits. In Claude Code, /context shows what is filling the window right now: system prompt, connected tools, instruction files, and conversation. /usage shows how much of your plan you have spent, and /cost totals the session on API billing. One look often changes the plan: a session that feels expensive because of long replies sometimes turns out to be carrying tens of thousands of tokens of connected tools it never uses.

Keep your instruction file thin

Your CLAUDE.md or AGENTS.md is reloaded into context every single session, so every line in it is a recurring cost. Keep it to cross-cutting rules, and move topical detail into reference files the agent reads only when needed. A fat instruction file taxes every session; a thin one with an index taxes only the sessions that use the detail.

Prune connectors and tools you are not using

The same thin-kernel rule applies to tooling. Every MCP server you connect ships its tool definitions into the context at session start, whether or not you ever call them, and an eagerly loaded skill costs the same way. Disconnect servers the current project does not use, and prefer skills that load on demand over rules pasted into the instruction file. Newer agent versions defer some tool schemas until they are needed, which softens the cost, but a leaner set of connectors still wins: /context will show you exactly what each one costs.

Match the model to the task

Not every step needs the largest model at the highest reasoning effort. Send hard design and debugging to the strong tier, and mechanical bulk work, renames, sweeps, and formatting, to a cheaper tier at reduced effort. Route per task, not per session: one session can mix tiers by handing the mechanical parts to subagents on the cheaper model.

The reasoning-effort dial works the same way: the default (high) is the sweet spot for most work. Save xhigh for the hardest coding and agentic tasks, and skip max almost always: past a point, extra thinking buys diminishing returns or outright overthinking while the token bill keeps climbing.

What a subagent really costs

A subagent starts with fresh context, rereads files your session already read, then spends more tokens reporting what it found back to you. Total spend usually goes up. What you buy is a short main transcript: a summary in place of thousands of lines of search output that every later turn would otherwise reread. Dispatch subagents for big, genuinely independent jobs; for a two-file lookup, a direct read is cheaper.

Workflows, for jobs that outgrow one context

The same economics extend to multi-agent workflows that fan a big job, an audit, a migration, a broad review, across many agents in parallel. Total spend goes up, main-session context stays small, and wall-clock time drops. Reach for workflows when the job cannot fit one context, not to hurry a small one.

Move repeatable work into scripts

An agent turn is the most expensive way to do anything a script could do. When a task is deterministic, the same transformation across many files, a data pull, a formatting sweep, have the agent write a script once and run the script from then on. Only the writing costs tokens; every later run is free. It is also more reliable: a script does not paraphrase, skip a file, or improvise.

Filter command output before the agent reads it

Shell output is the biggest silent cost: a large diff or test run can dump thousands of lines the agent never needed. Filtering it is the one technique on this site with full before-and-after logs: measured81.8% of output tokens removed across 93,690 commands (snapshot 2026-07-19). Per-command numbers and how it works are on the measurements page.

The tool behind those numbers is RTK, an open-source proxy that condenses command output before the agent reads it.

Frequently asked questions

Why do long agent sessions cost more tokens?

The model holds no memory between turns, so the whole conversation is resent and reprocessed on every request. A token written on turn one is read again on every later turn, so the cost of a long session compounds. Prompt caching can cut the billed price of that reread, but the tokens still occupy the context window and still dilute attention.

When should I use /clear, /compact, or a new session?

Use /clear when the next task shares nothing with the current conversation: it wipes context and costs the least. Use /compact at a natural checkpoint, after a fix lands and before a new area, when you need the conclusions but not the transcript. Start a new session when the task changes entirely; carrying an old transcript into new work makes every future turn pay for it.

How much does caveman mode save?

The target is roughly 50 percent of reply prose, deliberately lowballed: the logged caveman replies total 64.7 million tokens across 615 sessions, and the assumed 2x plain-prose baseline puts the saving at roughly 65 million tokens. The style compresses wording, not meaning; facts, caveats, code, and error strings are preserved exactly.

Do subagents and workflows save tokens?

Not by default. A subagent starts with fresh context, rereads the files your session already read, then spends more tokens reporting back, so total spend usually rises. What subagents buy is a short main transcript: a summary instead of thousands of lines of search output. Dispatch them for big, genuinely independent jobs, and do direct reads for small lookups.

What should I check first when I keep hitting the limit?

Audit before changing anything. Run /context to see what fills the window at session start, connected tools and instruction files are common surprises, and /usage to see plan spend. Then apply the fix that matches what you find: prune preloaded context if the session starts heavy, shorten and split sessions if the transcript is the weight, filter output if shell dumps dominate.
