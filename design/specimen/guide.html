Token discipline for AI coding agent sessions

savetokens.tips · application note AN-0001 · companion to datasheet ST-0001

Token discipline for AI coding agent sessions is the practice of keeping an agent's context window small on purpose, because every token in the conversation is resent and reprocessed on every turn. This note covers session hygiene from basic to advanced for Claude Code and Codex: clearing, compacting, session scoping, the caveman response style, model routing, and subagent economics.

Operating Principle

The model holds no memory between turns. Every request resends the whole conversation: your prompts, the agent's replies, and every line of tool output. A token written on turn one is read again on every later turn.

That makes session length multiplicative, not additive. Ten more turns on a bloated transcript cost far more than ten turns on a fresh one, because each turn pays for everything that came before it. Prompt caching can cut the billed price of the reread, but the tokens still occupy the window and still dilute the model's attention.

The working budget is finite: the context window holds [S]~200,0001 tokens. S, vendor spec, no counterfactual

  1. Vendor specification, not measured here, no counterfactual. A nominal ceiling for planning, never a characterized figure. Source: Anthropic, context windows documentation.

Note: stresses beyond these limits cause context to be dropped, not summarized. Sustained operation near maximum reduces answer quality. See datasheet ST-0001, Absolute Maximum Ratings.

Session Hygiene, Basic Practices

The three basic moves differ in how much context they drop. Choosing the right one is the whole skill.

/clear wipes the conversation and keeps the session open. Use it between unrelated tasks: nothing from the old task should tax the new one.

/compact replaces the transcript with a summary. Use it at a natural checkpoint, after a fix lands and before a new area, when you need the conclusions but not the word-for-word history. Compacting mid-investigation loses detail you still need, so finish the thought first.

A new session is the cleanest cut. When the task changes entirely, start fresh rather than steering a long transcript, because every future turn pays rent on everything already said.

STK-CAVEMAN-01, the Caveman Response Style

[E]~50%1
Reply prose reduction, design target
E, lowballed by design, no counterfactual

Caveman is a standing instruction that compresses the agent's reply prose while leaving every technical fact intact. The agent drops articles, filler, pleasantries, and hedging, speaks in fragments, and uses arrows for causality. Code, commits, file contents, function names, and error strings are never touched.

The rule that makes it safe is accuracy first, brevity second. The style compresses wording, not meaning: a fact, caveat, or qualifier is never dropped to save tokens, and the agent returns to plain prose for security warnings, irreversible-action confirmations, and any sequence where compression would risk a misread.

The target is deliberately lowballed at 50 percent. Caveman replies logged across 612 sessions total 64.5 million output tokens; assuming plain prose would have run 2x as long, the estimated saving is roughly 65 million tokens. The spent side is logged, the baseline is assumed, so the figure stays estimated.

Agent Installation, Copy Blocks

Both blocks below are agent-pasteable. Put them in the standing instruction file your agent reads every session: CLAUDE.md for Claude Code, AGENTS.md for Codex. Claude Code users can instead save the caveman block as a skill so it loads on demand.

STK-CAVEMAN-01 · response style block 
## Response style: caveman (ultra)

Respond terse like smart caveman. All technical substance stays; only filler goes.
Accuracy first, brevity second. Never drop a fact, caveat, or qualifier to save
tokens: compress wording, not meaning. If terse risks a misread, spend the words.

Drop: articles (a/an/the), filler (just/really/basically/actually/simply),
pleasantries (sure/certainly/of course), hedging. Fragments OK. Short synonyms
(big not extensive, fix not "implement a solution for"). Technical terms exact.
Code blocks unchanged. Error messages quoted exact.

Ultra intensity: abbreviate prose words (DB/auth/config/req/res/fn/impl), use
arrows for causality (X -> Y), one word when one word is enough. Code symbols,
function names, API names, and error strings are never abbreviated.

Auto-clarity: switch to plain prose for security warnings, irreversible-action
confirmations, and multi-step sequences where fragment order could be misread.
Resume caveman after the clear part is done.

Boundaries: code, commits, PR text, and file contents are written normally.
Off switch: "stop caveman" / "normal mode".

Intensity is tunable: delete the ultra paragraph for full, or keep articles and full sentences for lite (Section 3 table).

## Context discipline

- One task per session. Unrelated task -> /clear or a new session.
- Compact at natural checkpoints (after a fix lands, before a new area),
  never mid-investigation.
- Do not paste whole files into the prompt. Name the path and let the agent
  read targeted ranges.
- Keep this file a thin kernel. Topical detail lives in reference files the

- Subagents are not a discount. A subagent rereads context fresh and pays a

- Prefer filtered command output over raw dumps (diffs, logs, test runs).

Advanced Practices

STK-KERNEL-01, thin instruction kernel. The standing instruction file is reloaded into context every session, so every line in it is a recurring cost. Keep it to cross-cutting rules only, and move topical detail into reference files the agent reads on demand. A fat kernel taxes every session; a thin kernel with an index taxes only the sessions that need the detail.

STK-ROUTE-01, model and effort routing. Not every step needs the largest model at the highest reasoning effort. Route hard design and debugging to the strong tier, and mechanical bulk work, renames, sweeps, and formatting, to a cheaper tier at reduced effort. The route is per task, not per session, so one session can mix tiers through subagent dispatch.

STK-SUBAGENT-01, subagent dispatch discipline. A subagent is not a discount: it starts with fresh context, rereads files your session already read, then pays a summarize-back tax to report. What it buys is a thin main transcript, a short summary in place of thousands of lines of search output that every later turn would otherwise reread. Dispatch for large, genuinely independent scopes; for a two-file lookup, a direct read is cheaper.

Workflow orchestration. The same economics extend to multi-agent workflows that fan a big job, an audit, a migration, a broad review, across many agents in parallel. Total spend goes up, main-session context stays small, and wall-clock time drops. Reach for orchestration when the job cannot fit one context, not to speed up a small task.

Frequently Asked Questions

Why do long agent sessions cost more tokens?

The model holds no memory between turns, so the whole conversation is resent and reprocessed on every request. A token written on turn one is read again on every later turn, which makes the cost of a long session grow multiplicatively, not linearly. Prompt caching can cut the billed price of that reread, but the tokens still occupy the context window and still dilute attention.

When should I use /clear, /compact, or a new session?

Use /clear when the next task shares nothing with the current conversation: it wipes context and costs the least. Use /compact at a natural checkpoint, after a fix lands and before a new area, when you need the conclusions but not the transcript. Start a new session when the task changes entirely; carrying an old transcript into new work makes every future turn pay for it.

How much does the caveman response style save?

The design target is deliberately lowballed at roughly 50 percent of reply prose, and it is an estimate: no logged counterfactual exists, so this site prints it as [E] ~50 with a leading tilde and never in the measured accent color. Scaling the logged caveman reply tokens by the assumed 2x baseline puts the saving at roughly 65 million tokens across 612 sessions, an estimate, not a measurement. The style compresses wording, not meaning; facts, caveats, code, and error strings are preserved exactly.

Do subagents and workflows save tokens?

Not by default. A subagent starts with fresh context, rereads the files your session already read, then pays a summarize-back tax to report its findings, so total spend usually rises. What subagents buy is a thin main context: the transcript keeps a short summary instead of thousands of lines of search output. Dispatch them for large, genuinely independent scopes, and do direct reads for small lookups.
